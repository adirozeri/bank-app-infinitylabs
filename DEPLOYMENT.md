# BestBank — Production Deployment Guide

This document explains every step taken to deploy this app to Railway, why each step was needed, and whether it could have been avoided.

---

## Architecture Overview

- **Backend:** Node.js / Express v5 / TypeScript (`bestbank_be/`)
- **Frontend:** React / Vite / TypeScript (`bestbank_fe/`)
- **Database:** MongoDB Atlas (external, connection via `MONGO_URI`)
- **Hosting:** Railway — single service, backend serves the built frontend
- **Live URL:** https://backend-production-9ec8e.up.railway.app

---

## Step 1 — Explored the project structure

Read `Dockerfile`, `.env`, `package.json`, `server.ts`, `vite.config.ts`, `SocketContext.tsx`, and all route files.

**Why:** Before touching anything we needed to understand what the app does, what ports it uses, how the frontend talks to the backend (relative `/api/` paths), and what environment variables exist.

**Could it be avoided?** No — you can't deploy something you don't understand.

---

## Step 2 — Created `.gitignore` files

Added `fs/.gitignore` and `bestbank_fe/.gitignore` excluding `node_modules/`, `.env`, and `dist/`.

**Why:** Without these, `git add` would have committed 400MB of `node_modules` and all secrets (MongoDB password, API keys, email password) to a public GitHub repo.

**Could it be avoided?** No. Committing secrets to a public repo is a serious security issue.

---

## Step 3 — Deleted `node_modules/` to free disk space

Removed `bestbank_be/node_modules` (130MB) and `bestbank_fe/node_modules` (273MB).

**Why:** The disk was 100% full — no files could be written at all. `node_modules` is always regenerated by `npm install`, so deleting it loses nothing.

**Could it be avoided?** Yes — this was a machine-specific problem. If the disk had free space it wouldn't have been needed.

---

## Step 4 — Initialized a new git repo and pushed to GitHub

```bash
git init
git remote add origin https://github.com/adirozeri/bank-app-infinitylabs.git
git branch -m main
git add bestbank_be bestbank_fe .gitignore
git commit -m "Initial commit: BestBank full-stack app"
git push -u origin main
```

**Why:** The `fs/` folder was completely untracked — it lived inside the school's course repo (`git.infinitylabs.co.il`) but was never committed there. Railway deploys from GitHub, so a clean separate repo was needed for just the bank app.

**Could it be avoided?** No — Railway requires a GitHub repo to deploy from.

---

## Step 5 — Created the Railway project via GraphQL API

```bash
curl -X POST https://backboard.railway.com/graphql/v2 \
  -H "Authorization: Bearer <TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{ "query": "mutation { projectCreate(input: { name: \"bestbank\", workspaceId: \"<WORKSPACE_ID>\" }) { id name } }" }'
```

**Why:** Railway needs a project container before any services can exist. The Railway CLI was used via the GraphQL API because the CLI binary segfaults on this machine.

**Could it be avoided?** No, but normally you'd just run `railway init` in the terminal.

---

## Step 6 — Created the Railway service linked to GitHub

```bash
curl -X POST https://backboard.railway.com/graphql/v2 \
  -d '{ "query": "mutation { serviceCreate(input: { name: \"backend\", projectId: \"<PROJECT_ID>\", source: { repo: \"adirozeri/bank-app-infinitylabs\" }, branch: \"main\" }) { id } }" }'
```

Then configured the service instance:
- `rootDirectory`: `/` (repo root, so Docker can access both `bestbank_be/` and `bestbank_fe/`)
- `dockerfilePath`: `Dockerfile`

**Why:** This tells Railway which GitHub repo to watch and deploy from. The root directory was set to `/` (not `bestbank_be/`) because the multi-stage Dockerfile needs access to both frontend and backend folders.

**Could it be avoided?** No — it's the core deployment step.

---

## Step 7 — Set environment variables on Railway

Variables set on the service:

| Variable | Value |
|---|---|
| `MONGO_URI` | MongoDB Atlas connection string |
| `GEMINI_API_KEY` | Google Gemini API key |
| `EMAIL_USER` | Gmail address for sending OTP emails |
| `EMAIL_PASS` | Gmail app password |
| `EMAIL_SUPPLIER` | `gmail` |
| `OTP_EXPIRY_MINUTES` | `10` |
| `NODE_ENV` | `production` |
| `BASE_URL` | `https://backend-production-9ec8e.up.railway.app` |
| `CLIENT_ORIGIN` | `https://backend-production-9ec8e.up.railway.app` |

**Why:** The `.env` file was excluded from git (correctly — it contains secrets). Railway needs those values injected at runtime. Without `MONGO_URI` the server crashes immediately on startup. `CLIENT_ORIGIN` tells socket.io which domain is allowed to connect via WebSocket.

**Could it be avoided?** No. But `BASE_URL` and `CLIENT_ORIGIN` could reference Railway's built-in variable `${{RAILWAY_PUBLIC_DOMAIN}}` instead of being hardcoded — that's a cleaner approach.

---

## Step 8 — Generated a Railway domain

```bash
curl -X POST https://backboard.railway.com/graphql/v2 \
  -d '{ "query": "mutation { serviceDomainCreate(input: { serviceId: \"<SERVICE_ID>\", environmentId: \"<ENV_ID>\" }) { domain } }" }'
```

Result: `backend-production-9ec8e.up.railway.app`

**Why:** Needed the URL before setting `BASE_URL` and `CLIENT_ORIGIN` in the environment variables.

**Could it be avoided?** Yes — by using Railway's reference variable `${{RAILWAY_PUBLIC_DOMAIN}}` in the env vars instead of a hardcoded URL, this step becomes unnecessary.

---

## Step 9 — Created a root multi-stage Dockerfile

`Dockerfile` at the repo root:

```dockerfile
FROM node:20-alpine AS fe-builder
WORKDIR /fe
COPY bestbank_fe/package*.json ./
RUN npm ci
COPY bestbank_fe/ ./
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY bestbank_be/package*.json ./
RUN npm ci
COPY bestbank_be/ ./
COPY --from=fe-builder /fe/dist ./public
EXPOSE 3000
CMD ["npx", "tsx", "src/server.ts"]
```

**Why:** The frontend and backend are in separate folders. Docker's build context is scoped to one directory. A single Dockerfile at the repo root can access both. This implements the "backend serves the frontend" pattern — one Railway service, one URL, no CORS issues, and the frontend keeps its relative `/api/` paths without any changes.

**Could it be avoided?** Only if you switched the frontend to use an absolute `VITE_API_URL` env var (pointing to the backend's Railway URL) — then they could be two separate services. That would require changing every `axios.get('/api/...')` call. The single-service approach was simpler.

## Step 10 — Updated `server.ts` to serve frontend static files

```ts
import { join, dirname } from 'path';
import { fileURLToPath } from 'url';

const __dirname = dirname(fileURLToPath(import.meta.url));
const publicDir = join(__dirname, '../public');

app.use(express.static(publicDir));
app.get('/{*path}', (_req, res) => res.sendFile(join(publicDir, 'index.html')));
```

**Why:**
- `express.static` serves the built React files (JS, CSS, images).
- The catch-all route sends `index.html` for any unmatched path, so React Router handles client-side navigation. Without it, refreshing on `/dashboard` or `/register` returns a 404 from Express.

**Could it be avoided?** No — both lines are required to serve a built React SPA correctly.

## Bug Fix 1 — Express v5 wildcard syntax

**Error:** `PathError [TypeError]: Missing parameter name at index 1: *`

**Cause:** The project uses Express v5 (`"express": "^5.2.1"`). Express v5 changed the routing syntax — bare `*` wildcards are no longer valid.

**Fix:** Changed `app.get('*', ...)` to `app.get('/{*path}', ...)`.

**Could it be avoided?** Yes — by knowing about the Express v5 breaking change upfront, or by running a local Docker build before deploying.

---

## Bug Fix 2 — Wrong path to `public/index.html`

**Error:** `ENOENT: no such file or directory, stat '/public/index.html'`

**Cause:** `__dirname` at runtime is `/app/src`. Using `../../public` goes up two levels to `/` and then looks for `/public` — which doesn't exist. The correct path is one level up.

**Fix:** Changed `join(__dirname, '../../public')` to `join(__dirname, '../public')`.

**Path breakdown:**
```
/app/src   ← __dirname
/app       ← ../
/app/public ← ../public  ✓
```

**Could it be avoided?** Yes — by running `docker build . && docker run -p 3000:3000` locally before deploying, the path error would have shown up immediately.

---

## Lessons Learned

1. **Always do a local Docker build first** — `docker build . && docker run -p 3000:3000` catches path errors, missing files, and runtime crashes before they hit production.
2. **Know your Express version** — v5 has breaking changes from v4, especially in routing syntax.
3. **Use Railway reference variables** — `${{RAILWAY_PUBLIC_DOMAIN}}` avoids hardcoding URLs and the need to generate a domain before setting env vars.
4. **Keep disk space available** — full disk is a common blocker that wastes time.
5. **Separate secrets from code from day one** — `.gitignore` for `.env` should be the very first thing added to any project.
